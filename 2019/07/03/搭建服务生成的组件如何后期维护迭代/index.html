搭建服务生成的组件如何后期维护迭代

前言

正常情况下，搭建服务生成的页面或组件，或满足于快速产出试错，满足时效性，若涉及到频繁后期维护是需要独立开发的。但针对特殊场景，需要修改到元组件的数据结构，那么如何进行处理呢？以下根据一个实际场景来探索。

背景

店铺设计师开放平台提供设计师可以编辑拖拽模块生成装修皮肤的功能，比如现有如下真实需求（商品单行标题改为多行）

（期望效果） =>

在这个模块的设计规范中，价格和商品标题都支持拖拽（绝对布局），商品区块的整体高度，标题的高度都支持自定义。倘若只修改了商品标题行数，会有如下影响：

  • 整体区块带边框（左一）
  • 标题区块无固定高度（左二）
  • 价格区间绝对布局，导致展示异常（右二）
  • 标题高度设置不够，改为两行后展示不完整（最右）

emmm..唯一的解决办法那就是 洗库 ！

技术方案

大致方案分三步骤：校验 -> 更新数据 -> 测试

  1. 针对特定模块逻辑校验存储的模块皮肤数据的数据结构是否正确（存在脏数据以及多种设计规范的存在）
  2. 批量更新数据库操作
  3. 针对特定模块写自动化测试脚本 保证线上数据一致

模块皮肤数据存储格式

数据规范Gitlab

####初步方案
>

  • 校验存储的json数据结构是否符合当前设计规范
  • 校验标题节点[txt]和价格节点[cell]是否并列存在
  • 找到标题节点，在当前用户设置的高度基础上乘2，WebkitLineClamp属性设置为2
  • 找到价格节点，在当前的top属性下增加标题改变的单行高度
  • 找到父容器节点，设置高度为标题改变的单行高度

总结

  • 数据校验规则与清洗没有想象中容易，比如用户的骚操作，使得数据结构并不符合预期的规则。
  • 校验规则需要不断优化完善，其中还涉及到多次备份数据回滚，特别是线上库一定要慎重。
  • 数据校验规则并不能覆盖所有的场景，最优可能能达到95%-99%的准确率，剩余的特殊模块要手动操作。
  • UI测试脚本跑出来的截图发现问题也需要手动处理。
  • 日志记录异常数据非常重要，以免出现漏网之鱼。
